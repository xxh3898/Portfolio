2. 문제 해결 경험

CalmDesk 프로젝트에서 직원 대시보드 조회 시 응답 시간이 25초에 달하는 심각한 성능 문제를 직접 발견하고 해결했습니다.

문제 상황
직원 50명 규모의 사내 대시보드에서 목록 조회 시 약 25초의 응답 지연 발생. 서비스 운영 수준에서는 치명적인 사용성 저하였습니다.
문제 분석
SQL 쿼리 로그를 확인한 결과, for 루프 내에서 직원 1명당 출퇴근·기분·업무상태를 3개 레포지토리에 각각 단건 조회하는 구조였습니다. 직원 50명 × 쿼리 3개 = 총 150개의 쿼리가 순차 연쇄 발생하는 N+1 문제였습니다.
해결 방법
WHERE IN 절을 활용한 벌크 조회(단 3번의 쿼리로 전체 데이터 조회)로 전환하고, 조회 결과를 애플리케이션 메모리에서 Map<Long, Data> 구조로 변환해 O(1) 조회로 매핑하는 방식으로 리팩토링했습니다.
150개 → 3개
쿼리 호출 수 감소 (98%↓)
25초 → 0.9초
API 응답 속도 개선 (96%↑)
배운 점
성능 문제는 현상(느리다)이 아니라 구조(왜 느린가)에서 출발해야 한다는 것을 체감했습니다. 캐시 도입 같은 임시 처방이 아닌, 데이터 흐름 설계 자체를 바꾸는 방향으로 접근해야 지속 가능한 코드가 된다는 인사이트를 얻었습니다.
3. 협업 경험

CalmDesk 6인 팀 프로젝트에서 이슈 관리자 역할을 맡아 팀의 협업 체계를 설계하고 운영했습니다.

상황
초기 개발 단계에서 팀원들이 커밋 메시지·브랜치 네이밍을 제각각 사용해 코드 리뷰 및 변경 이력 추적이 어려웠습니다. PR에 설명이 부족해 리뷰어가 맥락을 파악하는 데도 시간이 많이 소요됐습니다.
본인의 행동
팀 전체에 공통 적용할 Git 커밋 컨벤션(feat / fix / refactor / docs 등)과 GitHub 이슈 템플릿(기능 구현 / 버그 리포트), PR 템플릿(변경 사항 / 테스트 결과 / 관련 이슈 번호)을 직접 작성하여 팀에 배포하고 사용법을 공유했습니다.
결과
PR 설명 템플릿 도입으로 리뷰 소요 시간을 평균 30% 이상 단축했으며, 팀장으로부터 "협업 체계 덕분에 후반 개발 일정 관리가 수월해졌다"는 피드백을 받았습니다. 최종 프로젝트 우수상 수상으로 이어졌습니다.
4. 입사 후 목표
단기 (1~2년)
빠르게 실제 서비스 코드베이스에 기여할 수 있는 개발자가 되는 것입니다. 코드 리뷰에 적극 참여하고, 기존 코드의 구조와 도메인 맥락을 깊이 이해해 첫 해에 실질적인 기능 개발과 버그 수정을 안정적으로 수행하겠습니다.
중기 (3~5년)
트래픽·데이터 규모가 커져도 안정적으로 동작하는 서버 시스템을 설계할 수 있는 백엔드 엔지니어로 성장하겠습니다. 성능 병목 분석, 쿼리 최적화, 서비스 분리 설계 등 실무 경험을 쌓아 팀 내에서 기술적 의사결정에 기여하겠습니다.
기여 방식
N+1 구조 개선처럼 성능 문제의 근본 원인을 데이터 흐름 관점에서 분석하고 구조적으로 해결하는 경험을 바탕으로, (회사명)의 백엔드 시스템 안정성과 응답 속도 개선에 직접 기여하겠습니다.
