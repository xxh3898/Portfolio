2. 문제 해결 경험

CalmDesk 프로젝트에서 직원 대시보드 조회 시 응답 시간이 25초에 달하는 심각한 성능 문제를 직접 발견하고 구조적으로 해결한 경험이 있습니다. 도입 초기, 직원 50명 규모의 사내 대시보드에서 목록 조회 시 약 25초의 응답 지연이 발생해 서비스 운영 수준에서는 치명적인 사용성 저하가 우려되었습니다.

원인을 파악하기 위해 SQL 쿼리 로그를 분석한 결과, for 루프 내에서 직원 1명당 출퇴근, 기분, 업무상태를 각각의 레포지토리에 단건으로 조회하는 방식이 원인이었습니다. 직원 50명에 대해 쿼리가 3개씩, 총 150개의 쿼리가 순차적으로 발생하는 전형적인 N+1 문제였습니다.

이를 해결하기 위해 WHERE IN 절을 활용한 벌크 조회 방식으로 전환하여, 단 3번의 쿼리만으로 전체 데이터를 불러오도록 리팩토링했습니다. 또한, 조회된 결과를 애플리케이션 메모리에서 Map<Long, Data> 구조로 변환하여 O(1) 조회가 가능하도록 로직을 최적화했습니다.

그 결과 150개에 달하던 쿼리 호출 수를 3개로 줄여 98% 감소시켰으며, 25초가 걸리던 API 응답 속도를 0.9초로 96% 단축할 수 있었습니다. 이 경험을 통해 성능 문제는 현상에 대한 임시 처방이 아니라, 데이터 흐름과 설계 구조 자체를 개선하는 방향으로 접근해야 지속 가능한 코드를 만들 수 있다는 점을 체감했습니다.

3. 협업 경험

CalmDesk 6인 팀 프로젝트에서는 이슈 관리자 역할을 담당하며 팀의 협업 체계를 주도적으로 설계하고 운영했습니다. 초기 개발 단계에서는 팀원들이 각기 다른 기준의 커밋 메시지와 브랜치 네이밍을 사용하여 변경 이력을 추적하거나 코드 리뷰를 진행하는 데 어려움이 많았습니다. 특히 PR에 설명이 부족해 리뷰어가 맥락을 파악하는 데 불필요한 시간이 소요되었습니다.

이러한 비효율을 개선하고자 팀 전체에 공통으로 적용할 수 있는 Git 커밋 컨벤션(feat, fix, refactor 등)을 명확히 세우고, 기능 구현과 버그 리포트를 위한 GitHub 이슈 템플릿과 PR 템플릿(변경 사항, 테스트 결과, 이슈 번호 포함)을 직접 제작하여 팀에 배포했습니다.

이처럼 체계화된 템플릿을 도입하고 지속적으로 문화를 정착시킨 결과, 리뷰 소요 시간을 평균 30% 이상 단축할 수 있었습니다. 또한 팀장으로부터 "명확한 협업 체계 덕분에 후반 개발 일정 관리가 수월해졌다"는 긍정적인 피드백을 받았으며, 궁극적으로 팀의 기여도를 높여 프로젝트 우수상 수상이라는 성과로 이어질 수 있었습니다.

4. 입사 후 목표
단기 (1~2년)
빠르게 실제 서비스 코드베이스에 기여할 수 있는 개발자가 되는 것입니다. 코드 리뷰에 적극 참여하고, 기존 코드의 구조와 도메인 맥락을 깊이 이해해 첫 해에 실질적인 기능 개발과 버그 수정을 안정적으로 수행하겠습니다.
중기 (3~5년)
트래픽·데이터 규모가 커져도 안정적으로 동작하는 서버 시스템을 설계할 수 있는 백엔드 엔지니어로 성장하겠습니다. 성능 병목 분석, 쿼리 최적화, 서비스 분리 설계 등 실무 경험을 쌓아 팀 내에서 기술적 의사결정에 기여하겠습니다.
기여 방식
N+1 구조 개선처럼 성능 문제의 근본 원인을 데이터 흐름 관점에서 분석하고 구조적으로 해결하는 경험을 바탕으로, (회사명)의 백엔드 시스템 안정성과 응답 속도 개선에 직접 기여하겠습니다.
