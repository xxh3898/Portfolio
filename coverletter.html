2. 문제 해결 경험

CalmDesk 프로젝트에서 관리자 대시보드 통계 조회 API(GET /api/dashboard/stats) 호출 시 25초에 달하는 지연을 발견하고, 애플리케이션 비효율이 클라우드 인프라에 미치는 영향을 분석해 근본적인 병목을 해결한 경험이 있습니다. Chrome 개발자 도구로 지연을 확인한 후 AWS Console을 점검해보니, EC2(T3 Micro) 인스턴스의 CPU 크레딧이 고갈되어 스로틀링 상태에 진입한 것을 파악했습니다.

과부하의 원인을 파악하기 위해 SQL 쿼리 로그를 분석한 결과, 직원 데이터 조회 중 발생한 N+1 쿼리 현상이 문제였습니다. 직원 50명 조회 시 150개의 단건 쿼리가 연쇄적으로 발생하면서 DB 연산을 폭증시키고 인스턴스의 CPU 자원을 과소비하고 있었습니다.

이를 해결하기 위해 JPA의 @Query JOIN FETCH를 활용해 연관 엔티티(Member)를 DB 계층에서 단일 벌크 쿼리(IN + JOIN FETCH) 1회로 한 번에 페치하는 구조로 전환했습니다. 이를 통해 150개의 단건 쿼리를 3개의 최적화된 벌크 쿼리로 통합하고, 애플리케이션 단에서는 이미 로딩된 엔티티를 기반으로 추가 DB 호출 없이 데이터를 조립했습니다.

그 결과 쿼리 호출 수를 150개에서 3개로 줄였고, 25초가 걸리던 API 응답 속도를 0.9초로 단축(96% 향상)했습니다. 더 중요한 것은 불필요한 연산 점유를 최소화함으로써 EC2 CPU 사용률을 정상화하고 CPU 크레딧 고갈 현상 방지라는 인프라 안정성까지 회복했다는 점입니다. 이 경험을 통해 단순한 코드 레벨의 비효율이 어떻게 클라우드 인프라 자원 마비로 직결될 수 있는지 체감하는 거시적인 병목 시야를 얻게 되었습니다.

3. 협업 경험

CalmDesk 6인 팀 프로젝트에서는 이슈 관리자 역할을 담당하며 팀의 협업 체계를 주도적으로 설계하고 운영했습니다. 초기 개발 단계에서는 팀원들이 각기 다른 기준의 커밋 메시지와 브랜치 네이밍을 사용하여 변경 이력을 추적하거나 코드 리뷰를 진행하는 데 어려움이 많았습니다. 특히 PR에 설명이 부족해 리뷰어가 맥락을 파악하는 데 불필요한 시간이 소요되었습니다.

이러한 비효율을 개선하고자 팀 전체에 공통으로 적용할 수 있는 Git 커밋 컨벤션(feat, fix, refactor 등)을 명확히 세우고, 기능 구현과 버그 리포트를 위한 GitHub 이슈 템플릿과 PR 템플릿(변경 사항, 테스트 결과, 이슈 번호 포함)을 직접 제작하여 팀에 배포했습니다.

이처럼 체계화된 템플릿을 도입하고 지속적으로 문화를 정착시킨 결과, 리뷰 소요 시간을 평균 30% 이상 단축할 수 있었습니다. 또한 팀장으로부터 "명확한 협업 체계 덕분에 후반 개발 일정 관리가 수월해졌다"는 긍정적인 피드백을 받았으며, 궁극적으로 팀의 기여도를 높여 프로젝트 우수상 수상이라는 성과로 이어질 수 있었습니다.
